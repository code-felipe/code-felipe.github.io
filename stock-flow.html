Why I built this

SupplyFlow taught me how to think in layers — controllers, services, repositories. But it was a full-stack monolith, tightly coupled to Thymeleaf and a browser. When I started StockFlow, I had one goal: strip all of that away and learn what it really means to design a backend that speaks JSON.

No views. No form submissions. Just HTTP, resources, and contracts. StockFlow is where I learned to think like an API.

What it solves

Any product-driven system — e-commerce, inventory management, catalog tools — needs a reliable backend layer to handle:

  • Managing thousands of products without losing track of what's active vs. deleted
  • Filtering catalogs dynamically: by name, price range, or stock availability
  • Keeping inventory records separate from product data but always in sync
  • Returning clean, predictable JSON responses that any frontend can consume

StockFlow answers those needs with a structured REST API — clean endpoints, validated inputs, layered architecture, and soft deletion that preserves history without cluttering the active catalog.

What I learned building it

REST API Design

Learning what "RESTful" actually means in practice

I'd heard the term REST API a hundred times. Building one from scratch is when you understand why it matters. Every decision — URL structure, HTTP verbs, status codes, response shape — is a contract with whoever calls your API.

I used @RestController with ResponseEntity wrappers to control exactly what comes back — the status code, the body, the shape. A 200 OK with an empty list is not the same as a 404 Not Found, even if both "have no data."

Method Endpoint Description
GET /api/products List all active products (paginated)
GET /api/products/{id} Get a single product by ID
POST /api/products Create a new product
PUT /api/products/{id} Update an existing product
DELETE /api/products/{id} Soft-delete a product
GET /api/products/search Filter by name, price, stock range 

@GetMapping("/{id}")
public ResponseEntity<ProductResponseDTO> getById(@PathVariable Long id) {
    return productService.findActiveById(id)
        .map(ResponseEntity::ok)
        .orElse(ResponseEntity.notFound().build());
}

@PostMapping
public ResponseEntity<ProductResponseDTO> create(
        @Valid @RequestBody ProductRequestDTO dto) {
    ProductResponseDTO created = productService.create(dto);
    URI location = URI.create("/api/products/" + created.getId());
    return ResponseEntity.created(location).body(created);
}

Dynamic Filtering with Specifications

Building search that actually scales

A hardcoded query works until someone adds a fourth filter parameter. I needed something composable — filters you can stack or skip depending on what the client sends.

The solution was JPA Specification. Each filter (name, price range, stock range) is its own predicate. They get combined at query time only when the parameter is present. The result: one endpoint, flexible behavior, no conditional query explosion.

"I had to understand LEFT JOIN vs INNER JOIN in JPA, because products without inventory records would silently disappear from results. A LEFT JOIN keeps them visible — returning null for stock instead of dropping the row entirely." 


// ProductSpecification.java
public class ProductSpecification {

    public static Specification<Product> nameLike(String name) {
        return (root, query, cb) ->
            name == null ? null :
            cb.like(cb.lower(root.get("name")),
                    "%" + name.toLowerCase() + "%");
    }

    public static Specification<Product> priceBetween(
            BigDecimal min, BigDecimal max) {
        return (root, query, cb) -> {
            if (min == null && max == null) return null;
            if (min == null) return cb.lessThanOrEqualTo(root.get("price"), max);
            if (max == null) return cb.greaterThanOrEqualTo(root.get("price"), min);
            return cb.between(root.get("price"), min, max);
        };
    }

    public static Specification<Product> stockBetween(
            Integer minStock, Integer maxStock) {
        return (root, query, cb) -> {
            Join<Product, Inventory> inv =
                root.join("inventory", JoinType.LEFT);
            if (minStock == null && maxStock == null) return null;
            if (minStock == null)
                return cb.lessThanOrEqualTo(inv.get("quantity"), maxStock);
            if (maxStock == null)
                return cb.greaterThanOrEqualTo(inv.get("quantity"), minStock);
            return cb.between(inv.get("quantity"), minStock, maxStock);
        };
    }
}

Domain Model & DTOs

Separating what you store from what you expose

In SupplyFlow I built every DTO by hand, constructor and all. In StockFlow I used Lombok — a single @Builder annotation replaces 40+ lines of boilerplate. But more importantly, I understood why the separation exists: your database model and your API contract are different things and they should stay that way.

Product

  • id Long
  • name String
  • sku String
  • description String
  • price BigDecimal
  • isActive Boolean
  • createdAt LocalDateTime
  • inventory Inventory

Inventory

  • id Long
  • quantity Integer
  • updatedAt LocalDateTime
  • product Product

// Lombok replaces all the boilerplate from SupplyFlow
@Data
@Builder
@NoArgsConstructor
@AllArgsConstructor
public class ProductResponseDTO {
    private Long id;
    private String name;
    private String sku;
    private BigDecimal price;
    private Integer stock;       // flattened from Inventory
    private Boolean isActive;
    private LocalDateTime createdAt;
}

// Mapper: entity → DTO
public ProductResponseDTO toDTO(Product p) {
    Integer stock = (p.getInventory() != null)
        ? p.getInventory().getQuantity() : null;

    return ProductResponseDTO.builder()
        .id(p.getId())
        .name(p.getName())
        .sku(p.getSku())
        .price(p.getPrice())
        .stock(stock)
        .isActive(p.getIsActive())
        .createdAt(p.getCreatedAt())
        .build();
}

Soft Deletion

Delete without losing history

A real DELETE removes data forever. In most business systems, that's the wrong move — you might need to audit past orders, reference a deactivated product, or simply undo a mistake. Soft deletion solves this with a flag instead of a row removal.

Every product has an isActive boolean. "Deleting" a product sets it to false. The repository always filters by isActive = true by default — so deactivated records become invisible to the catalog without being gone from the database. 


// Repository: only return active products
@Query("SELECT p FROM Product p WHERE p.isActive = true")
Page<Product> findAllActive(Pageable pageable);

// Service: soft delete
@Override
@Transactional
public void softDelete(Long id) {
    Product product = productDao.findById(id)
        .orElseThrow(() -> new ResourceNotFoundException(
            "Product not found: " + id));

    product.setIsActive(false);
    productDao.save(product);
}

Validation & Exception Handling

Controlling what breaks and how it breaks

In a REST API, the response is the UI. When something goes wrong, the error body is the only feedback the client gets. I used Bean Validation (@NotNull, @NotBlank, @Positive) on request DTOs and a centralized @ControllerAdvice to convert every exception into a consistent JSON structure.

"Before this, any unhandled exception returned a giant Spring HTML error page. After the global handler, every error — validation failure, not found, server error — returns the same clean JSON shape with a proper status code." 


// Request DTO with validation annotations
@Data
@NoArgsConstructor
@AllArgsConstructor
public class ProductRequestDTO {

    @NotBlank(message = "Name is required")
    private String name;

    @NotBlank(message = "SKU is required")
    private String sku;

    @NotNull(message = "Price is required")
    @Positive(message = "Price must be greater than zero")
    private BigDecimal price;

    private String description;
}

// Global exception handler
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<ErrorResponse> handleValidation(
            MethodArgumentNotValidException ex) {
        List<String> errors = ex.getBindingResult()
            .getFieldErrors().stream()
            .map(e -> e.getField() + ": " + e.getDefaultMessage())
            .toList();

        return ResponseEntity.badRequest()
            .body(new ErrorResponse(400, "Validation failed", errors));
    }

    @ExceptionHandler(ResourceNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleNotFound(
            ResourceNotFoundException ex) {
        return ResponseEntity.status(404)
            .body(new ErrorResponse(404, ex.getMessage(), null));
    }
}

Swagger / OpenAPI Documentation

An API without documentation is a black box

One thing you don't need to worry about in a Thymeleaf app: the form tells the user what fields exist. In a REST API, nothing is visible unless you document it. I integrated SpringDoc OpenAPI so every endpoint, parameter, and response schema is auto-generated and browsable at /swagger-ui.html.

It also made testing faster — instead of writing curl commands by hand during development, I could test directly from the browser with real request examples. Documentation and dev tool in one. 


// pom.xml dependency
<dependency>
    <groupId>org.springdoc</groupId>
    <artifactId>springdoc-openapi-starter-webmvc-ui</artifactId>
    <version>2.3.0</version>
</dependency>

// Annotating the controller for richer docs
@Operation(summary = "Get all active products",
           description = "Returns a paginated list of active products")
@ApiResponse(responseCode = "200", description = "OK")
@GetMapping
public ResponseEntity<Page<ProductResponseDTO>> getAll(
        @ParameterObject Pageable pageable) {
    return ResponseEntity.ok(productService.findAllActive(pageable));
}

What I'd do differently

  • JWT Authentication from the start. Security is currently in the roadmap. Adding it mid-project means touching a lot of files that were already written without it in mind.
  • Integration tests, not just manual Swagger testing. Swagger is fast during development, but a proper test suite with MockMvc would have been more reliable for catching regressions.
  • Stricter DTO separation earlier. I reused some DTOs across create and update operations. They're subtly different enough that they should have been separate from day one.

The bigger picture

SupplyFlow taught me how to build a system. StockFlow is teaching me how to design one.
